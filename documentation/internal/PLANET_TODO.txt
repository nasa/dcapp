Planet Extension - TODO and Notes
==================================

1. Resource Conflicts (multi-planet rendering)
----------------------------------------------

Rendering two planet instances per frame causes artifacts on the first
planet. Root cause: shared global state in plPlanetContext (gptCtx) in
pl_planet_ext.c.

Known issues:

  tCurrentDynamicBufferBlock (FIXED)
    Was on global plPlanetContext. Second render() call overwrote the
    first planet's dynamic uniform data (MVP matrices, texture indices,
    flags). Fixed by moving to a local variable in pl_render_planet and
    passing through pl__render_chunk as a parameter.

  tStagingBuffer
    Shared staging buffer written at hardcoded offset 0. Both planets'
    prepare() calls memcpy to the same location. Second prepare()
    overwrites the first planet's data before the GPU blit executes.
    Fix: per-planet staging buffers or sub-allocation with different
    offsets per prepare() call.

  Shared vertex/index buffers
    Single vertex buffer, single index buffer, single free list for
    each. Eviction logic walks per-planet replacement queues, but freed
    memory goes back to the shared pool. One planet can starve the
    other under memory pressure.

  Shared bindless texture system
    tTextureIndexHashmap and atBindGroups are global. If one planet
    evicts a chunk whose texture index is still referenced by an
    in-flight draw from another planet, the slot could be reused,
    causing the wrong texture to be sampled.

  Shared scratch buffers
    sbcScratchBuffer and sbcScratchBuffer2 are used during chunk
    load/unload. Interleaved operations across two planets could
    clobber string data.


2. PlanetTexture XML Support
----------------------------

Planned XML element for overlaying a texture on the planet surface:

  <PlanetTexture Source="path/to/texture.png" .../>

As a child of <Planet>, similar to <PlanetData>.

Metadata (lat/lon center, meters per pixel) source is TBD:
  - XML attributes directly on the element
  - Embedded in the PNG file
  - A separate .json file alongside the PNG

Current state:
  - Element type DC_APP_ELEM_TYPE_PLANET_TEXTURE is registered
  - Stub handler _process_xml_node_planet_texture exists in
    _dcapp_process_xml.c (logs Source attribute but doesn't store it)

Remaining work:
  - Add texture fields to _NodePlanet in dcapp.h
  - Fill in the stub to parse and store all attributes
  - Pass texture data to plPlanetInit.atTextures[0] in _init_planets


3. Dynamic Texture Features (future)
-------------------------------------

Requirements:
  - Load a texture that may not exist yet at startup
  - Reload/replace texture during runtime when file changes on disk
  - Change texture position (lat/lon) at runtime
  - Change resolution (meters per pixel) at runtime
  - Texture may need to be repositioned (moved) at runtime

Current limitation:
  The planet extension bakes textures into .pak files during
  create_planet(). There is no public API to update textures after
  creation. The plPlanetI vtable has no set_texture or update_texture
  function.

Planned approach:
  - Add new public API to plPlanetI (e.g. update_texture)
  - Re-run texture splitting logic (reuse code from pl__planet_load)
  - Re-pack affected .pak files
  - Force-evict currently-resident textured chunks
  - Let residency system naturally reload with new texture data
